## 2026-04-02 — Phrase Boundary Audit and Full Address Correction

### Problem identification

**Word 215 "Acknowledge":** Audio sounded like a 5-frame fragment (~180 ms) instead of the expected ~800 ms. Investigation showed the CSV end address (0x1C9E) was only 46 bytes into the phrase; the actual STOP frame and voice EPROM index boundary were at 0x1D1A (170 bytes).

**Word 982 "Good Evening":** Export was silently skipped. Two-part root cause:
1. CSV had `start = end = 0xC42C` (0 bytes).
2. The `LPC_DATA_END` constant in `cat-1000_lpc_export.py` was set to `0xC42C` (= start of the last phrase), hard-capping the scan window to 0 bytes.

### Root cause of systematic truncation

Original CSV end addresses were generated by scanning for STOP frames using the wrong bit direction (LSB-first equivalent), which found false STOP frames at earlier positions than the real ones. This affected the majority of the 481-phrase vocabulary.

### Calibration of correct bit direction

Using the voice EPROM index as the authoritative phrase boundary, calibrated the MSB-native STOP scanner:
- Word 220 "Affirmative": EPROM index = 180 bytes, MSB scanner = 179 bytes (off by 1, systematic).
- Word 981 "Good Afternoon": 195 bytes, scanner = 195 bytes. 
- Word 982 "Good Evening": 149 bytes, scanner = 149 bytes. 

The 1-byte off-by-1 in the scanner is systematic (`bytes_consumed()` rounds the final partially-consumed byte up) and harmless — the EPROM index is used for the authoritative end address.

### Fixes applied

| Fix | Detail |
|-----|--------|
| Word 215 end address | `0x1C9E` → `0x1D1A` in both CSVs |
| Word 982 end address | `0xC42C` → `0xC4C1` in both CSVs |
| `LPC_DATA_END` constant | `0xC42C` → `0xC4C1` in `cat-1000_lpc_export.py` |

Verification: word 215 exported as 170 B / 800 ms ; word 982 exported as 149 B / 675 ms .

### Full audit

Compared all 481 CSV end addresses against voice EPROM index boundaries:

- **Initial audit:** 272 words had end addresses shorter than the EPROM index end (truncated). Delta distribution: 1–5 bytes (16), 6–15 bytes (23), 16–50 bytes (107), 51–200 bytes (124), 201–1000 bytes (2).
- **Identified 9 discrepant words** where MSB-native STOP was significantly before the index end: words 17, 260, 344, 348, 476, 602, 663, 732, 831. These were skipped in the first bulk fix pass.
- **Identified duplicate EPROM entry problem:** Word 600 had two entries in the EPROM index (phrase_id=0 appears twice in group 6). The bulk fix script used a word-number key, so the second entry's index_end (0x907A) overwrote the first entry's correct end (0x6CAE), producing a 9,256-byte range for "Miles." Word 10 "Ten" had the same issue.

### Bulk fix — pass 1 (word-number keyed)

Script `fix_csv_addresses.py` built `word → index_end` dict and updated all 272 truncated entries (plus 10 more found in the updated count). Fixed 282 entries, skipped 9 discrepant.

**Post-fix issues found:**
- Word 600 "Miles": `0x6C26–0x907A` (wrong; 9 KB). Corrected to `0x6C26–0x6CAE` (136 B).
- Word 600 duplicate entry `0x9041–0x90DC` removed from clips CSV.
- Verified word 600 exports as 136 B / 625 ms and sounds like "Miles." 

### Bulk fix — pass 2 (start-address keyed)

Second pass using per-start-address `correct_end` dict found 183 additional entries with incorrect end addresses (both over-extended and remaining truncated, including the 9 previously-skipped discrepant words). All corrected.

**Final state:** All 481 entries in both CSVs exactly match `index_end(start_addr)`. Zero mismatches.

## 2026-04-03 — CAT-310DX Reverse Engineering

### Session overview

Full reverse engineering of the CAT-310DX V1.00 unified EPROM (`CAT-310DX_V1-00_1998_7A69.BIN`). Parallel goals: extract all LPC speech clips, confirm codec compatibility, add license headers, generate .gitignore, add lesspipe note to CAT-1000 docs, update all documentation.

### Hardware identification

- CPU confirmed: Intel 8052 (MCS-51). Timer2 inline ISR at 0x002B (CPL P3.4; CLR 0xC6; RETI) proves MCS-51 variant is 8052.
- Speech synthesizer: TSP53C30 (TMS5220-compatible) on-board, mapped at MOVX 0xC000. Discovered via disassembly of streaming loop at 0x4A78 (`MOVC A,@A+DPTR` → `MOVX @DPTR,A` to 0xC000).
- LPC data: 41,827 bytes at 0x4FA0–0xF6E3 (previously believed absent).

### eprom_scout false positives

Running `eprom_scout.py` on the full image produced 364 false LPC blobs and a false phrase index at 0x1098 (a CJNE keypad dispatch table). Constrained scan to 0x4FA0–0xF6E3 found 58 legitimate blobs, but phrase-index detection still failed — word boundaries are defined by the dispatch table at 0x4AC3, not a self-contained phrase index.

### Dispatch table analysis (0x4AC3–0x4F9F)

Parsed all 415 × 3-byte entries `[word_id, addr_hi, addr_lo]`:
- 100 word IDs (0x00–0x63 = numbers 0–99, hex value = decimal number)
- 414 unique ROM addresses; words 0x00 and 0x34 share address 0x4FA0
- 96 of 100 word IDs have 4–9 recordings for concatenative intonation variation
- Total speech data: 41,820 bytes

### Bit direction confirmation

Synthesized Set 1 clips in both orientations using `synth_310dx.py --both`:
- MSB-first: consistent 400–850 ms durations, normal STOP frame detection
- LSB-first: many 5000 ms timeouts (no STOP frame found), unintelligible audio
- **MSB-first confirmed correct**, matching CAT-1000 format

### Codec compatibility confirmed

`render_phrase_to_pcm()` from `cat-1000_lpc_export.py` processes CAT-310DX clips without modification. Same TMS5220 LPC-10 bit-packed frame format on both devices.

### Full extraction

`extract_310dx.py` script created: parses dispatch table, derives 414 clip boundaries, writes `.lpc` and `.wav` to `all_310dx_clips/`. Results:
- 414 unique clips extracted and synthesized
- Word 99 (last clip) hit synthesis timeout — minor boundary artifact
- 413 other clips clean
